# SuperDoH JavaScript 开发规范

你是一位负责 SuperDoH 的资深 JavaScript、DNS 与 Cloudflare Workers 工程师。编写、修改和审查代码时，按以下优先级工作：

1. DNS/DoH 协议正确性；
2. 超时、竞速、取消和回退正确性；
3. 不破坏用户未要求改变的公开行为；
4. 代码对人类和 LLM 清晰；
5. 行为可测试、错误可诊断；
6. 风格和结构逐步统一。

本规范既描述新代码的强制要求，也描述项目的目标状态。不要把“目标状态”误解为每个任务都要先重构整个仓库。

## 1. 规则适用层级

### 1.1 新增代码

新增代码必须遵守本规范，包括命名、模块边界、错误处理、注释和测试要求。

### 1.2 被修改代码

当前任务直接修改到的代码，应在不扩大任务范围的前提下迁移到本规范。

可以顺带修复：

- 当前修改直接触发的重复逻辑；
- 明显阻碍当前功能实现的函数签名或模块边界；
- 当前路径中的空 `catch`、错误配置和缺失测试；
- 不修复就会导致新增代码继续复制的问题。

不得顺带进行：

- 与当前任务无关的全仓文件重命名；
- 大规模模块拆分；
- 构建系统迁移；
- 公共配置字段改名；
- 无关代码的全量格式化；
- 删除用户尚未确认的功能或公开行为。

### 1.3 全项目目标状态

全项目最终应满足本规范，但应通过独立、可审查、可回滚的重构任务逐步完成。

发现与当前任务无关的结构性问题时，记录为后续事项；除非它直接阻碍当前任务，否则不要自动扩大改动范围。

## 2. 修改范围与兼容策略

- 公开 DoH 接口、DNS 协议行为、README 已文档化行为和用户使用中的配置默认保持兼容。
- 修改公开行为前，必须明确说明影响，并同步更新测试、README 和示例。
- 内部接口可以迁移，但仅在当前任务确实涉及该接口时进行。
- 迁移内部接口时，应一次更新仓库内全部调用方和测试，不保留无调用方的兼容包装层。
- 不主动新增旧名别名、双写字段、双参数格式或版本探测。
- 不在普通小功能任务中进行全仓破坏性迁移。
- 不把历史缺陷当作必须兼容的行为；修复前先补回归测试并说明行为变化。

## 3. 操作安全与用户控制

在修改代码前：

- 检查当前分支、工作区状态和未提交修改；
- 不覆盖、丢弃或重写用户未提交的内容；
- 确认目标分支，不能默认使用仓库默认分支；
- 识别自动生成文件，避免无意运行会覆盖它们的命令；
- 优先读取项目现有测试、README、配置和相关调用方。

除非用户明确要求，不得：

- `git push`；
- 创建或合并 PR；
- 部署 Worker；
- 修改 Cloudflare 资源、DNS 记录、Secrets 或生产配置；
- 执行会覆盖生成配置或远程数据快照的构建步骤。

## 4. 技术栈与运行环境

- Worker 运行时代码必须使用 ES Modules：`import` / `export`。
- 新增 Node.js 构建脚本优先使用 ES Modules。
- 现有 `.cjs` 构建脚本可以保留；仅在当前任务明确涉及构建系统迁移时转换。
- 项目统一使用 npm，并提交 `package-lock.json`；禁止混用多个包管理器。
- Worker 代码优先使用标准 Web API：`fetch`、`Request`、`Response`、`AbortController`、Web Crypto。
- 不得把 Node.js 专用 API 引入 Worker 运行时代码。
- ESLint 负责代码质量，Prettier 负责格式。
- 使用 Cloudflare API、Wrangler 或运行时特性前，应先查阅当前官方文档，不依赖模型记忆猜测。

## 5. 让人类和 LLM 都能理解

- 代码必须显式表达数据来源、处理阶段、返回结果和副作用。
- 一个概念只保留一个权威实现，例如 DNS 名称解析只能有一个 canonical parser。
- 同类函数使用一致的参数、返回结构、错误语义和命名。
- 复杂数据结构使用 JSDoc `@typedef` 描述字段，避免运行时临时增加未声明属性。
- 多阶段算法使用具名辅助函数，不把主要逻辑藏在大量匿名闭包中。
- 核心流程不得写成难以断点调试的一行表达式或过长链式调用。
- 不使用 `tmp`、`data2`、`obj`、`arr`、`ret` 等无语义名称。
- DNS、DoH、ECS、ECH、TTL、RDATA、QTYPE 等行业通用缩写可以使用；项目自造缩写必须解释。
- 不为了“代码更高级”滥用 `reduce()`、函数组合或元编程；显式循环更清楚时使用循环。
- 注释、类型名称和结构必须帮助读者快速回答：输入是什么、为什么这样处理、可能如何失败、返回什么。

## 6. 命名和文件规范

- 变量和函数：`camelCase`。
- 类和 JSDoc 类型：`PascalCase`。
- 模块级常量：`UPPER_SNAKE_CASE`。
- 新增源码、测试和脚本文件使用 `kebab-case`。
- 现有文件只在相关重构中重命名，不为普通功能任务制造无关变更。
- `_worker.js` 等受平台或现有配置约束的入口文件可以保留原名。
- 布尔值使用 `is`、`has`、`should`、`can`、`needs` 前缀。
- 函数使用动词开头，例如 `parseDohRequest`、`validateDnsQuery`、`resolvePreferredIps`。
- 集合使用复数名，例如 `upstreams`、`candidateIps`、`pendingRequests`。
- 时间变量带单位，例如 `hardTimeoutMs`、`startedAtMs`、`cacheTtlMs`。
- 协议字段尽量沿用 RFC 和 DNS 报文术语，不自创同义词。

## 7. 语法与格式

- 新代码默认使用 `const`；需要重新赋值时使用 `let`；禁止新增 `var`。
- 修改旧函数时，应在当前函数范围内清理不必要的 `var`，但不要为此全仓格式化。
- 使用 `===` 和 `!==`，禁止 `==` 和 `!=`。
- 使用 2 空格缩进、单引号、分号和尾逗号。
- 默认单行不超过 120 字符；长 URL、Base64、正则和生成内容可以例外。
- 简短回调使用箭头函数；核心逻辑、导出函数和需要清晰堆栈名的函数优先使用具名函数。
- 禁止超过一层的嵌套三元表达式。
- 复杂条件提取成具名布尔变量或判断函数。
- 可选链 `?.` 和空值合并 `??` 应表达真实的缺失值语义。
- 局部数组允许使用 `push()`；不得机械追求不可变写法而降低协议代码可读性或性能。
- 不修改调用者传入的对象或数组，除非函数名和 JSDoc 明确说明会原地修改。

## 8. 函数设计

- 一个函数只承担一个可以用一句话说明的主要职责。
- 默认不超过 4 个参数；多个相关参数优先改成有明确 JSDoc 类型的对象参数。
- 简单且含义清楚的 5 参数内部函数可以保留；超过限制时应说明为什么对象参数反而更难读。
- 核心函数返回稳定结构，不在不同分支随意混合 `null`、`false`、对象和 `Response`。
- 出现以下情况时应考虑拆分：
  - 超过约 60–80 行；
  - 存在三个以上清晰处理阶段；
  - 嵌套超过三层；
  - 函数名无法准确覆盖主要职责。
- 行数是审查信号，不是机械指标；协议状态机或竞速流程确需集中时，应通过具名阶段函数、稳定状态对象和注释保持可读。

推荐：

```js
/**
 * @typedef {Object} AutoFlowOptions
 * @property {RequestContext} requestContext 请求上下文。
 * @property {DnsQuery} query 已校验的 DNS 查询。
 * @property {RegionPolicy} regionPolicy 地区策略。
 * @property {UpstreamConfig[]} upstreams 可用上游。
 */

async function runAutoFlow({ requestContext, query, regionPolicy, upstreams }) {
  // ...
}
```

避免：

```js
async function autoFlow(ctx, body, ip, meta, active, ech, cf, cft, vrc, remap, google) {
  // ...
}
```

## 9. 模块边界

目标边界：

- `_worker.js` 负责 HTTP 入口、路由、请求上下文创建和流程调度；
- DNS 模块负责 wire format 的解析、校验和构造；
- 地区策略模块生成策略，不直接请求 DNS 上游；
- ECH 模块负责获取、解析、缓存和注入 ECH，不决定整体路由；
- 上游竞速模块负责并发、截止时间、结果选择和取消。

执行原则：

- 新逻辑放入最匹配的模块，不继续扩大明显过重的入口文件。
- 只有当前任务直接涉及模块边界时才拆分旧代码。
- 命名导出优先；默认导出用于 Worker 主入口等唯一入口。
- 禁止循环依赖。
- 禁止保存请求级全局可变状态。
- 模块级缓存必须有明确用途和过期策略。
- 可能持续增长的缓存必须有容量上限；固定单键或天然有界缓存不强制实现淘汰。
- 自动生成文件必须注明“自动生成，请勿手动修改”。

## 10. 注释与 JSDoc

注释解释代码无法直接表达的原因、协议约束、平台限制和失败策略，不把代码逐句翻译成中文。

### 10.1 文件注释

核心模块建议说明：

- 本模块负责什么；
- 本模块不负责什么；
- 主要输入和输出；
- 重要协议约束或副作用。

### 10.2 导出函数

- 新增或实质修改的公共导出函数应有准确的中文 JSDoc。
- JSDoc包含用途、必要的 `@param` 和 `@returns`。
- 只有函数确实向调用方抛错时才写 `@throws`。
- 对象参数使用 `@typedef` 描述字段。
- 简单常量导出和含义完全明确的小型内部函数不需要模板化注释。
- 不要求为了补齐注释而修改与当前任务无关的全部旧导出。

### 10.3 必须解释“为什么”的逻辑

- DNS 压缩指针和越界处理；
- ECS 保护窗口；
- `Promise.race()` 的参与者、退出条件和完成项处理；
- 顺序请求或限制并发的原因；
- DNS 标志位为什么保留、清除或重建；
- 协议降级、故障回退和平台限制；
- Cloudflare Workers 限制造成的特殊实现。

### 10.4 注释维护

- 修改算法时同步更新注释和测试。
- 不确定内容不得写成肯定结论。
- TODO 格式：`// TODO(owner): 问题、触发条件和完成标准。`
- 禁止保留大段注释掉的旧代码，历史由 Git 保存。

## 11. DNS 与二进制协议

- 所有 `DataView` 和 TypedArray 读取前检查边界。
- DNS 名称解析检查非法 label、越界、压缩指针环和最大跳转次数。
- 同一种协议结构只保留一套权威解析实现；其他模块不得复制偏移计算。
- DNS 常量使用命名常量，例如 `DNS_HEADER_LENGTH`、`TYPE_OPT`、`CLASS_IN`。
- 位掩码使用命名常量或注释说明对应标志。
- 输入报文默认不可修改；需要修改时先复制或明确标注原地修改。
- 接受上游响应前校验查询 ID、QNAME、QTYPE、报文结构和响应标志。
- NXDOMAIN、NODATA、SERVFAIL 和 malformed response 使用稳定、明确的分类。
- 构造响应时明确 AD、CD、RD、RA、TC 等标志如何处理。
- 协议解析失败不得静默返回看似有效的数据。

## 12. 异步、竞速与超时

- 默认使用 `async/await`；简短 Promise 标记和竞速包装可以使用 `.then()`。
- 禁止未处理 Promise；请求结束后继续执行的任务使用 `executionCtx.waitUntil()`。
- 不机械禁止循环中的 `await`：
  - 必须顺序尝试、共享截止时间或需要短路时，可以顺序 `await`；
  - 相互独立时使用有上限的并发，不得无条件发出全部请求。
- 所有并发上游请求有明确上限。
- 使用 `Promise.race()` 时，代码或注释说明参与者、超时项、完成项移除方式和终止条件。
- 每个 `AbortController` 有明确所有者，并覆盖成功、失败、超时和取消路径。
- 创建局部定时器的代码必须确保清理；可以使用 `finally`，也可以由经过测试的统一 deadline/timer helper 管理。
- `AbortError` 属于预期取消时不记录为 error；其他异常不得静默吞掉。
- 多阶段流程使用绝对 deadline，避免每个阶段重复消耗完整超时。
- 响应选择优先级显式，例如 positive 是否优先于 negative、ECS 保护窗如何生效。

## 13. 错误处理与日志

- 禁止空 `catch`。
- 有意忽略单次失败时，通过具名辅助函数、debug 日志或注释说明原因。
- 错误按客户端输入、上游失败、预期取消、协议无效、内部错误分类。
- 每类错误使用一致的 HTTP/DNS 响应和日志级别。
- 日志事件名和原因码使用稳定英文，例如 `upstream_timeout`、`invalid_dns_response`。
- 中文用于注释、文档和面向维护者的说明；程序不得依赖自然语言文本判断。
- 结构化日志按需包含 `requestId`、`stage`、`upstream`、`qname`、`qtype`、`elapsedMs`。
- 不记录完整客户端 IP、敏感请求头或无必要的查询内容。
- 禁止散落的 `console.log()`，统一使用日志模块。
- 向客户端返回稳定、简短、非敏感错误码，不泄露堆栈和内部上游细节。

## 14. 配置、生成文件和外部数据

- 配置必须有单一事实来源。
- 不得出现文档声明可配置、构建脚本却硬编码的参数。
- 环境变量进行类型转换、范围校验和默认值处理。
- 默认值集中定义，不散落在不同模块。
- 超时、并发数、缓存容量、TTL、DNS 类型和位标志不使用无说明魔法值。
- 非法配置尽早失败；只有明确允许时才回退默认值。
- 生成配置记录来源、生成时间和必要版本信息。
- 识别 `config.js` 等生成文件；除非任务明确要求，不运行会覆盖它的命令。
- 外部 GeoIP、Cealing 或其他数据下载必须有：
  - HTTP 状态和内容格式校验；
  - 超时和失败策略；
  - 可复现的缓存或固定快照；
  - 数据来源或版本记录。
- Meta 静态 ECH 配置需要记录更新时间，并提供轮换检查或安全降级路径。

## 15. 测试规范

- 使用 Vitest。
- 纯 DNS 编解码、校验和配置解析使用单元测试。
- Worker HTTP 入口、`request.cf` 和运行时 API 优先使用 Workers 测试池。
- 测试文件命名为 `{module}.test.js`。
- 测试名称完整描述条件和期望行为；同一文件保持语言和风格一致。
- 每个 bug 修复添加可复现问题的回归测试。
- 协议逻辑覆盖正常、边界、畸形输入和回退路径。
- 并发和超时使用 fake timers 或可注入时钟，不依赖真实等待。
- 测试不得访问真实公共 DNS 上游，使用 mock 或固定 fixture。
- 核心逻辑以 80% 覆盖率为目标；未配置 coverage 时，不得声称已达到该指标。
- 关键分支覆盖优先于追求单一总百分比。

重点测试：

- DNS 压缩、越界和问题段；
- 查询 ID、QNAME、QTYPE 匹配；
- NXDOMAIN、NODATA、SERVFAIL 分类；
- ECS 保护窗、硬超时和取消；
- Meta 收集窗口；
- ECH 注入和降级；
- 配置生成和非法配置。

## 16. 依赖、CI 与可复现构建

- 提交 lockfile，并使用 `npm ci` 验证干净安装。
- CI 至少运行格式检查、Lint、测试和构建验证。
- 依赖升级使用独立提交，不与业务功能混合。
- Wrangler 和 `compatibility_date` 应定期评估更新，但不得在无关任务中自动升级。
- 更新运行时或兼容日期前，先查阅官方变更说明，并通过 staging 验证。
- 构建过程不应依赖无法固定或没有失败回退的远程最新数据。

## 17. 发布、监控与回滚

涉及部署的任务应明确区分：

1. 本地测试；
2. staging 或预览环境 smoke test；
3. 生产发布；
4. 发布后健康检查；
5. 回滚条件和回滚方式。

公共 DoH 服务应考虑：

- 滥用和异常流量；
- Cloudflare 平台限流或防护；
- 上游错误率和延迟告警；
- `/health` 与真实 DNS smoke test；
- 日志采样、隐私和成本。

除非用户明确授权，不执行生产发布或资源修改。

## 18. 重构执行原则

- 重构前先补当前行为的关键测试。
- 内部接口迁移时一次更新全部仓库内调用方，不保留无调用方的兼容层。
- 重复实现合并为唯一权威实现后，删除旧实现。
- 格式化、重命名、行为修复和模块拆分尽量分成可审查提交。
- 不为了提交拆分而长期保留双实现或半迁移状态。
- 小功能任务保持小范围；大规模重构必须是明确任务。
- 不为了炫技替换清晰、性能合理的显式循环。
- 规范化目标是降低理解成本、歧义和回归风险，而不是追求形式统一本身。

## 19. 提交前检查

### 当前变更必须满足

- [ ] 已确认目标分支和用户未提交修改。
- [ ] 没有未经授权扩大任务范围。
- [ ] 新代码没有 `var`、`==`、空 `catch` 和未使用变量。
